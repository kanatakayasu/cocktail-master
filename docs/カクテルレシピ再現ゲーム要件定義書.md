# カクテルレシピ再現ゲーム 要件定義書

## 1. 文書概要

### 1-1. 目的

既存の「カクテルマスター」は、図鑑・クイズ・暗記カード・飲料分類によって知識を学ぶ構成になっている。  
本要件では、既存の学習データを活用しつつ、**カクテルレシピを実際の作業手順として再現する学習用ゲーム**をWebアプリケーション内に追加する。

このゲームの目的は以下の通り。

- 材料、製法、バーツール、作業順の理解を知識から実践へ移す
- 受験勉強で覚えたカクテルレシピを、操作を通じて定着させる
- 間違えた箇所をログ比較で可視化し、復習効率を上げる
- 既存アプリの学習体験に「反復練習」と「ゲーム性」を加える

### 1-2. 対象ユーザー

- カクテルアドバイザー資格を学習中のユーザー
- カクテル初心者で、レシピの流れを体感的に覚えたいユーザー
- バーツールや技法を視覚的に理解したいユーザー

### 1-3. スコープ

本要件の対象は、既存Webアプリ内に追加する**フロントエンド完結型の学習ゲーム機能**である。

対象に含むもの:

- 新規ゲームページ
- 注文生成、調製操作、採点、復習フィードバック
- 学習進捗のローカル保存
- バー背景演出、操作アニメーション
- デフォルメ調イラストアセット

対象に含まないもの:

- バックエンドAPI
- ログイン、ランキング、マルチプレイ
- 課金、広告
- 音声ナレーション必須化

## 2. 既存資産の活用前提

### 2-1. 既存アプリ構成

現行アプリは以下の構成を持つ。

- 技術スタック: React 19 + TypeScript + Vite + Tailwind CSS + Framer Motion
- 既存ページ: ホーム、図鑑、クイズ、暗記カード、飲料分類
- データ管理: `src/data/` 配下のTypeScriptデータ
- 進捗保存: LocalStorage 前提

### 2-2. 再利用する既存データ

| データ | 既存ファイル | 再利用内容 |
|------|------|------|
| カクテルレシピ | `src/data/cocktails.ts` | カクテル名、材料、技法、グラス、ガーニッシュ、説明 |
| バーツール | `src/data/tools.ts` | 道具名、用途、アイコン表現 |
| 技法 | `src/data/techniques.ts` | ビルド / ステア / シェイク / ブレンド等の手順説明 |
| グラス | `src/data/glasses.ts` | 提供グラス情報 |
| スピリッツ / リキュール | `src/data/spirits.ts`, `src/data/liqueurs.ts` | 酒棚のカテゴリ構成 |
| 飲料分類 | `src/data/beverageTree.ts` | 酒棚UIの分類構造、材料グルーピング |

### 2-3. 既存データの不足点

現行の `CocktailRecipe` は学習表示には十分だが、ゲーム採点には不足がある。主な不足点は以下。

- 材料ごとの**正規化ID**がない
- 「ライム」と「ライムジュース」のような**棚配置上の差**が未整理
- 各レシピに対して**操作手順の正解データ**がない
- 計量、氷、注ぐ順、ストレインなどの**作業単位**が定義されていない
- 現状60レシピあるが、`blend` は実質 `ピニャ・コラーダ` 1件のみで、技法バランスに偏りがある

そのため、ゲーム実装時には既存レシピとは別に**ゲーム専用の正解手順データ**を持つ。

## 3. ゲームコンセプト

### 3-1. 基本コンセプト

- 舞台はバーのカウンター
- 注文票が飛んでくる
- プレイヤーは各棚からアイテムを選び、カウンターへ持ち込む
- カウンター上に並んだアイテムだけを使って調製する
- 適切なバーツールを使い、適切な製法を適切な順番で行う
- 完成後に採点し、どこで間違えたかをログ比較で返す

### 3-2. 空間構成コンセプト

プレイ画面は「移動と作業」の体験を持つ。  
カウンターを拠点として、以下の棚からアイテムを取り寄せて調理を行う。

- 酒棚
- 冷蔵庫
- グラス棚
- バーツール棚
- カウンター（調理エリア）

### 3-3. 学習価値

このゲームで定着させる学習対象は以下。

- カクテル名と使用材料の対応
- ベーススピリッツと副材料の組み合わせ
- ビルド / ステア / シェイク / ブレンドの使い分け
- 使用する主ツールと補助ツール
- レシピの進行順序
- ミスの種類ごとの原因把握

## 4. ゲームの到達目標

### 4-1. 学習目標

- ユーザーが1杯のカクテルを、ヒントありで再現できる
- ユーザーが間違いを「材料」「道具」「技法」「順番」で認識できる
- ユーザーが繰り返しプレイにより正答率を上げられる

### 4-2. プロダクト目標

- 既存学習アプリに自然に統合される
- 1プレイが短く、反復しやすい
- バーの雰囲気と学習UIの両立ができる
- モバイルでも操作が破綻しない

## 5. プレイフロー

1. ホームまたはナビゲーションからゲームページへ遷移する
2. モードと難易度を選択する
3. 注文票にカクテル名が表示される
4. 酒棚、冷蔵庫、グラス棚、バーツール棚から必要なアイテムを選択する
5. 選択したアイテムをカウンターへ移動させ、作業可能状態にする
6. カウンター上で手順どおりに調製する
7. 「提供」操作で作業を確定する
8. システムが作業ログと正解データを比較する
9. 結果画面で点数、ミス内容、正しい手順を表示する
10. リトライまたは次の注文へ進む

## 6. ゲームモード要件

### 6-1. 必須モード

#### チュートリアルモード

- 初回ユーザー向け
- 次にやるべき操作を強くガイドする
- 間違い操作前に警告を出せる
- 採点は緩めでよい

#### 練習モード

- 学習用の基本モード
- 材料、主ツール、製法、フェーズ順を判定対象にする
- 間違えた場合は結果画面で詳細フィードバックを返す

#### チャレンジモード

- 実戦形式
- ヒントなし
- より厳密な順番・分量・グラス・ガーニッシュ判定を行う
- タイムまたは連続正解数をスコアに反映できる

### 6-2. 難易度別の判定粒度

| 項目 | チュートリアル | 練習 | チャレンジ |
|------|------|------|------|
| 材料選択 | 必須 | 必須 | 必須 |
| 主ツール選択 | 必須 | 必須 | 必須 |
| 製法 | 必須 | 必須 | 必須 |
| 順番 | フェーズ単位で判定 | フェーズ内も判定 | 厳密判定 |
| 分量 | 任意 | 一部判定 | 厳密判定 |
| グラス | 任意 | 任意または加点 | 必須 |
| ガーニッシュ | 任意 | 任意または加点 | 必須 |
| タイム評価 | なし | 任意 | あり |

## 7. 機能要件

### FR-01. ゲーム導線

- ホーム画面にゲーム導線を追加する
- ルーティングにゲームページを追加する
- 初回表示時にゲーム概要と操作説明を表示する

推奨ルート:

- `/game`
- `/game/play`
- `/game/result`

### FR-02. 注文生成

- 対応済みレシピからランダムまたは選択式で注文を生成する
- 注文票には最低限以下を表示する
  - カクテル名
  - 難易度
  - ヒント利用可否
- チュートリアルでは補助情報を表示できる
  - ベーススピリッツ
  - 使用技法
  - 使用グラス

### FR-03. 酒棚・材料選択

- プレイヤーは棚単位でアイテムを選択できる
- 対象棚は `酒棚 / 冷蔵庫 / グラス棚 / バーツール棚` とする
- 酒棚カテゴリは既存の `spirits`, `liqueurs`, `beverageTree` を基礎に構成する
- プレイヤーはアイテムをクリックまたはタップで選択できる
- モバイルではドラッグ必須にせず、タップ操作でも完結できる
- 間違った材料も選択可能とし、その操作はログに残す

棚ごとの最低限カテゴリ:

- 酒棚: ベーススピリッツ、リキュール
- 冷蔵庫: ジュース、炭酸飲料、シロップ、乳製品、フルーツ / ガーニッシュ
- グラス棚: カクテルグラス、タンブラー、コリンズ等
- バーツール棚: シェーカー、ジガー、バースプーン、ミキシンググラス等

### FR-03A. カウンター搬入（拠点）機能

- カウンターはゲーム中の作業拠点とする
- 棚から選択したアイテムは、カウンターへ移動して初めて使用可能とする
- カウンターには「現在持ち込み中の材料・グラス・ツール」を表示する
- 持ち込み済みアイテムは状態を区別する
  - 未使用
  - 使用中
  - 使用済み
- 不要アイテムを持ち込んだ場合もログに残し、採点対象にする

### FR-04. 分量選択

- 練習モード以上では、必要に応じて分量を選択または計量できる
- `45ml`, `30ml`, `15ml`, `1tsp`, `適量`, `1カット` などを正規化して扱う
- `適量` は厳密mlではなく、操作種別として判定する

補足:

- MVPでは「材料選択のみ正誤判定、分量はチャレンジのみ厳密化」でもよい
- ただしデータ構造としては最初から分量フィールドを持つ

### FR-05. バーツール選択

- プレイヤーはツールトレイから適切な道具を選択できる
- 少なくとも以下の主ツールをゲーム対象に含める
  - シェーカー
  - ミキシンググラス
  - バースプーン
  - ストレーナー
  - ジガー
  - ブレンダー
- 必要に応じて補助ツールも採点対象にできる
  - スクイーザー
  - ペストル
  - マドラー

### FR-06. 調製操作

ゲームは操作を「工程」として扱う。  
棚からカウンターへの搬入も工程として記録する。最低限以下の工程タイプを持つ。

- 棚を開く
- アイテムを選択する
- カウンターへ移動する

- グラスを選ぶ
- 氷を入れる
- 材料を量る
- 材料を注ぐ
- ツールを選ぶ
- シェイクする
- ステアする
- ブレンドする
- ストレインする
- 炭酸で満たす
- ガーニッシュを添える
- 提供する

### FR-07. 技法別インタラクション

#### ビルド

- グラス上で直接組み立てる
- 炭酸系は軽いステア表現を持つ

#### ステア

- ミキシンググラスとバースプーンを使用する
- 必要に応じてストレーナーで提供グラスへ注ぐ

#### シェイク

- シェーカーを選んだ後、短い操作アクションを行う
- 操作方法はクリック連打、スワイプ、長押しのいずれかでよい

#### ブレンド

- ブレンダーを選んで起動し、完了後にグラスへ注ぐ

### FR-08. 作業ログ記録

すべての操作を時系列でログ化する。

ログに残す最低項目:

- 操作時刻
- 操作種別
- 対象アイテムID
- 操作元エリア（酒棚 / 冷蔵庫 / グラス棚 / バーツール棚 / カウンター）
- 操作先エリア
- 対象カテゴリ
- 使用ツールID
- 分量
- 工程番号

### FR-09. 採点・判定

ゲームは、プレイヤーの作業ログを正解データと比較し、少なくとも以下を判定する。

- 正しい材料を選んだか
- 不足材料がないか
- 不要な材料を入れていないか
- 正しい主ツールを選んだか
- 正しい製法を使ったか
- フェーズ順または工程順が正しいか
- 提供まで完了したか

追加判定項目:

- 分量が正しいか
- グラスが正しいか
- ガーニッシュが正しいか
- 補助ツールの使い方が正しいか

### FR-10. ミス分析フィードバック

結果画面では、単なる正誤だけでなく、**どこで間違えたか**を明示する。

表示要件:

- 総合評価（Perfect / Good / Retry など）
- 点数
- 正解 / 不正解の内訳
- 最初の誤りが発生した工程
- プレイヤーの作業ログ
- 正しい手順ログ
- ミスのカテゴリ別指摘

ミスカテゴリの例:

- 材料ミス
- 分量ミス
- ツールミス
- 製法ミス
- 順番ミス
- ガーニッシュミス
- 未完了

### FR-11. リトライと復習導線

- 同じ注文をすぐに再挑戦できる
- 正しいレシピを確認してから再挑戦できる
- 結果画面から図鑑の該当カクテル詳細へ遷移できる

### FR-12. 学習進捗保存

- 以下を LocalStorage に保存する
  - プレイ履歴
  - カクテルごとの最高評価
  - 苦手カテゴリ
  - 連続正解数
  - モード別成績

### FR-13. ヒント機能

- チュートリアル / 練習モードではヒントを表示できる
- ヒントは段階的に開示する
  - ベースを表示
  - 使用技法を表示
  - 次の工程を表示
  - 完全手順を表示
- ヒント使用時はスコアを減点できる

## 8. 判定ロジック要件

### 8-1. 判定の考え方

レシピの正解は「単なる材料一覧」ではなく、**工程列**として定義する。  
そのうえで、プレイヤーの `actionLog` と正解 `answerSteps` を比較する。

### 8-2. 工程の粒度

各レシピは最低でも以下の単位で正解工程を持つ。

- `selectGlass`
- `addIce`
- `measureIngredient`
- `addIngredient`
- `selectTool`
- `performTechnique`
- `strain`
- `topUp`
- `garnish`
- `serve`

### 8-3. 順番判定

順番判定はモードに応じて切り替える。

- チュートリアル: 大きなフェーズ順のみ判定
- 練習: フェーズ内の主要工程順を判定
- チャレンジ: 工程単位で厳密に判定

フェーズ例:

1. 準備
2. 計量
3. 混合
4. 仕上げ
5. 提供

### 8-4. 許容ルール

厳密すぎる判定は学習阻害になるため、以下のような許容を持たせる。

- 同一フェーズ内の一部工程は入れ替え許容
- `適量` は厳密分量でなく「正しい材料の追加」で判定
- ガーニッシュは練習モードでは減点対象、チャレンジでは必須
- 補助ツールはチュートリアルでは不問にできる

### 8-5. スコア配分例

| 項目 | 配点例 |
|------|------|
| 材料正解 | 40 |
| 主ツール正解 | 15 |
| 製法正解 | 20 |
| 順番正解 | 15 |
| 提供完了 | 10 |

チャレンジ追加配点:

- 分量正解: 10
- グラス正解: 5
- ガーニッシュ正解: 5
- 速度ボーナス: 5

## 9. データ要件

### 9-1. 新規に必要なデータモデル

既存の `CocktailRecipe` とは別に、ゲーム専用データを追加する。

推奨追加データ:

- `src/data/ingredientsCatalog.ts`
- `src/data/gameRecipes.ts`
- `src/data/gameModes.ts`

### 9-2. 推奨型定義

```ts
type GameActionType =
  | 'openShelf'
  | 'pickItem'
  | 'moveToCounter'
  | 'selectGlass'
  | 'addIce'
  | 'measureIngredient'
  | 'addIngredient'
  | 'selectTool'
  | 'performTechnique'
  | 'strain'
  | 'topUp'
  | 'garnish'
  | 'serve';

interface IngredientMaster {
  id: string;
  label: string;
  categoryId: string;
  aliases: string[];
  isAlcohol: boolean;
  assetId?: string;
}

interface GameRecipeStep {
  id: string;
  phase: 'prep' | 'measure' | 'mix' | 'finish' | 'serve';
  actionType: GameActionType;
  targetId?: string;
  toolId?: string;
  amount?: string;
  strictOrder: boolean;
  optional?: boolean;
  hint?: string;
}

interface GameRecipe {
  id: string;
  cocktailId: string;
  difficulty: 1 | 2 | 3;
  allowedModes: ('tutorial' | 'practice' | 'challenge')[];
  primaryTechnique: 'build' | 'stir' | 'shake' | 'blend';
  steps: GameRecipeStep[];
}

interface GameActionLog {
  actionType: GameActionType;
  targetId?: string;
  toolId?: string;
  amount?: string;
  fromArea?: 'liquorShelf' | 'fridge' | 'glassShelf' | 'toolShelf' | 'counter';
  toArea?: 'counter' | 'mixingGlass' | 'shaker' | 'glass';
  timestamp: number;
}
```

### 9-3. データ設計上の重要要件

- カクテル表示用データとゲーム採点用データは分離する
- 材料名はIDベースで正規化する
- 別名や表記ゆれは `aliases` で吸収する
- ゲーム用データはレシピごとに手順を持つ
- 将来的にノンアルコールやホットカクテルも追加可能な構造にする

## 10. 画面要件

### 10-1. 画面一覧

必須画面:

- ゲームトップ画面
- プレイ画面
- 結果画面
- 進捗 / 成績画面

### 10-2. ゲームトップ画面

表示要件:

- モード選択
- 難易度選択
- 対応カクテル数
- 学習ポイント説明
- 初回チュートリアル導線

### 10-3. プレイ画面

主要UI領域:

- 背景バー
- 注文票
- 酒棚
- 冷蔵庫
- グラス棚
- バーツール棚
- カウンター（拠点）
- 現在の選択状態表示
- 持ち込み済みアイテム表示
- 作業ログ表示
- ヒントボタン
- 提供ボタン

ビジュアル要件:

- 各棚（酒棚 / 冷蔵庫 / グラス棚 / バーツール棚）はデフォルメイラスト背景で描画する
- カウンター拠点もデフォルメイラスト背景で描画する

- デスクトップ: 左に各棚、中央にカウンター、右に注文票とログ

表示要件:

- 総合ランク

- リトライ

- 図鑑へ戻る

### 10-5. 進捗画面

表示要件:

- カクテルごとの到達度
- 技法別正答率
- 苦手材料 / 苦手技法
- 最近のプレイ履歴

## 11. UI / UX要件

### 11-1. 操作方針

- PCではクリック中心、必要ならドラッグを補助的に使う
- モバイルではタップ中心で完結する
- 誤操作しやすい長押し必須UIは避ける
- 選択中の材料・道具・工程は常に明示する

### 11-2. 学習しやすさ

- 誤答しても即終了にせず、学びに変換する
- 緊張感よりも反復しやすさを優先する
- 情報量は多いが、1画面で迷わない配置にする

### 11-3. 世界観

- 既存アプリのダークなバー世界観を引き継ぐ
- ネオンではなく、落ち着いたアンバー系の照明を主軸にする
- 学習アプリとして文字の視認性を優先する

## 12. アニメーション要件

### 12-1. 方針

アニメーションは見た目のためだけでなく、**操作の意味を伝える演出**として使う。

### 12-2. 必須アニメーション

| 対象 | 内容 |
|------|------|
| バー背景 | 微細な光、ボトル棚の揺らぎ、空気感の演出 |
| 注文票 | 上から滑り込む、またはカウンターに置かれる演出 |
| 棚UI | 棚の開閉、選択棚のフォーカス遷移 |
| 材料選択 | ボトルが持ち上がる、選択時に縁が光る |
| 棚→カウンター移動 | 選択アイテムがカウンターへ移動し、並ぶ演出 |
| カウンター整列 | 持ち込みアイテムがカテゴリごとに並び直される演出 |
| 注ぐ操作 | 液体がグラスやシェーカーへ流れる演出 |
| シェイク | シェーカーの往復運動、霜エフェクト |
| ステア | バースプーンの回転、液面の揺れ |
| ブレンド | ブレンダー起動時の振動と液体変化 |
| 成功判定 | スタンプ、光、軽い祝福演出 |
| 失敗判定 | 揺れ、赤みのフィードバック、ミス箇所強調 |

### 12-3. 調理アニメーション詳細要件

- 「選択」だけで終わらず、カウンター上での調理動作が見えることを必須とする
- 各製法で最低1つの明確な主動作を持つ
  - ビルド: ボトル傾き + 液体注入 + 軽いステア
  - ステア: ミキシンググラス内の回転流体 + スプーン回転
  - シェイク: シェーカー往復 + 霜発生 + 仕上げ注ぎ
  - ブレンド: ブレンダー振動 + 液面変化 + 注ぎ
- 調理中は対象ツールと容器を強調表示し、どこで何をしているか視覚的に分かること

### 12-4. 実装要件

- 基本は `Framer Motion` で実装する
- 重い連続演算は避ける
- `prefers-reduced-motion` に対応する
- プレイ中の主要操作は0.2〜0.6秒程度でレスポンス良く返す

## 13. イラスト要件

### 13-1. アートディレクション

- デフォルメ調
- 親しみやすい
- バーの雰囲気を壊さない
- 写実ではなく、学習UIに馴染む記号化された表現
- 子ども向けに寄りすぎず、大人の学習アプリとして成立するトーン
- 棚、カウンター、アイテム、背景すべてを同一テイストのデフォルメで統一する

### 13-2. 画風要件

- 2Dイラスト
- 少し丸みのあるシルエット
- ボトルやグラスは識別しやすい簡略化
- 色数は多すぎず、カテゴリで見分けやすくする
- 線の太さ、陰影、ハイライトルールを統一する
- 棚やカウンター背景にも同じ線・陰影ルールを適用し、浮いた素材感を出さない

### 13-3. 必須アセット一覧

| アセット | 用途 | 推奨形式 |
|------|------|------|
| バー背景イラスト | プレイ画面のベース背景 | SVG または WebP |
| 酒棚パーツ | 棚レイアウトとカテゴリ差分 | SVG |
| 冷蔵庫パーツ | 扉、棚段、冷気演出用 | SVG |
| グラス棚パーツ | 複数グラスの見せ分け用 | SVG |
| バーツール棚パーツ | ツールの掛け置き表現用 | SVG |
| カウンター天板パーツ | 調理拠点のUIベース | SVG |
| ボトルイラスト | スピリッツ、リキュール、副材料 | SVG |
| グラスイラスト | 提供グラス表示 | SVG |
| ツールイラスト | シェーカー、ジガー等 | SVG |
| 注文票イラスト | オーダーUI | SVG |
| ガーニッシュ素材 | ライム、レモン、チェリー等 | SVG |
| 演出素材 | 霜、液体、光、判定スタンプ | SVG / PNG |

### 13-4. アセット状態

各アセットは最低限以下の状態を持つ。

- 通常
- ホバーまたは選択可能
- 選択中
- 使用済み
- 不正解時の強調

### 13-5. 背景・棚・アイテム統一ルール

- 「背景だけ写実」「アイテムだけフラット」などの混在は禁止する
- 棚、カウンター、各アイテムは同一イラストセットとして制作する
- 棚背景は装飾ではなく、操作対象エリアとして判別しやすい形状にする
- カウンター背景は作業位置が分かるガイド付きデフォルメイラストにする

## 14. 非機能要件

### 14-1. 性能

- 主要操作後の視覚応答は100ms以内に始まること
- 一般的なノートPCとスマートフォンで快適に動作すること
- 画像アセットは初期表示を阻害しないよう最適化すること

### 14-2. レスポンシブ

- 360px幅程度のスマートフォン表示に対応する
- タッチターゲットは十分な大きさを確保する

### 14-3. アクセシビリティ

- キーボード操作でも主要操作に到達できること
- 色だけで状態を伝えないこと
- モーション低減設定を尊重すること

### 14-4. 保守性

- データとUIと判定ロジックを分離する
- 採点ロジックは関数化し、テスト可能にする
- 追加レシピ対応時に既存コードの分岐が増えすぎない構造にする

## 15. MVP要件

初期実装で必須とする範囲は以下。

- ゲームページ新設
- 注文 -> 調製 -> 提供 -> 結果 の一連フロー
- チュートリアル / 練習 / チャレンジの3モード
- 材料、主ツール、技法、順番の判定
- 作業ログと正解手順の比較表示
- LocalStorage による成績保存
- バー背景、主要ツール、ボトル、グラスの基本イラスト
- 主要操作アニメーション

MVP対象レシピ数の推奨:

- まずは12〜16レシピ
- `build`, `stir`, `shake`, `blend` を最低1件以上含める

## 16. 将来拡張要件

- 全60レシピ対応
- 日替わりチャレンジ
- 苦手レシピ自動出題
- 実績バッジ
- 効果音、環境音
- お客様キャラクター追加
- ストーリーモード
- ノンアルコール / ホットカクテル拡張

## 17. 受け入れ条件

以下を満たした場合、本要件を満たすMVPとみなす。

- ユーザーがホームからゲームへ遷移できる
- 1件の注文を最後までプレイできる
- 棚から選択したアイテムがカウンターへ移動し、視覚的に並ぶ
- カウンター上のアイテムのみを使って調理できる
- 間違った材料を選ぶと、結果画面で材料ミスとして表示される
- 間違った製法を選ぶと、結果画面で製法ミスとして表示される
- 順番が違う場合、結果画面で順番ミスとして表示される
- 自分の手順ログと正しい手順ログを比較できる
- 再挑戦ができる
- 成績がリロード後も保持される
- モバイル表示でプレイ不能にならない
- イラストとアニメーションが最低限ゲーム全体に適用されている

## 18. 実装上の懸念点

- 材料表記の正規化が先に必要
- 現行データだけでは工程粒度が足りないため、手動整備コストが発生する
- 分量厳密判定はデータ整備なしでは破綻しやすい
- ブレンド技法の対応レシピが少ないため、演出の費用対効果を見極める必要がある
- モバイルでの棚UIは情報量過多になりやすく、カテゴリ切り替え設計が重要
